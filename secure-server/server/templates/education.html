How We Secured Our Website

We've worked to implement features that address all of the relevant vulnerabilities on the OWASP Top 10, as well as other common web-related attacks.

Common Web Vulnerabilities:

Broken access controls come from a failure to correctly implement authorization, leading to attackers being able to access pages they should not have the privilege to do. This could be by modifying URLs to direct to authenticated pages, forcing a browser to visit them, or by other means.

We prevent this by ensuring that any pages intended for logged-in users require a valid session to visit, and only implementing access control policies in a single file (to avoid misconfiguration).

Leaked data is often the result of cryptographic failures. This category is rather broad, and can come from data being unprotected in tranist, cryptographic secrets being leaked in source files, deprecated algorithms being used, to name a few reasons.

Our website requires HTTPS to view, and has a valid certificate to facilitate this. User passwords are hashed and salted with bcrypt, which aligns with modern password storage standards. Secrets are inaccessible to external viewing.

Injection attacks take place when user input fields aren't validated or sanitized correctly.

We prevent inejctions because none of our URLs are parameterized, and all of our input fields whitelist only particular characters, leaving out opportunities for injection with things like script tags and SQL injections.

Many languages and libraries have the ability to configure security settings--however, the defaults often aren't the most secure option.

All of the standard security headers are enabled for the browser. There are no extraneous files anywhere, nor are there overly informative error messages.

Software and data integrity failures come about when the security of third party libraries and the develpment pipeline aren't mandated.

Our project is relatively small, therefore integrity of our repository and contributors is easy to maintain. We also ensure that all of our dependencies are up-to-date before we publish the server.

Logs and monitoring are necessary for detecting current attacks and trying to prevent future ones, and many sites don't have these necessary logging capabilities.

While we don't have logs stored anywhere presently, an output feed can be seen on our server. Additionally, we monitor for standard attacks (account timeout, rate limiting, DDoS prevention).

Web applications are vulnerable to SSRF when attackers can cause the server to access pages the end user shouldn't have access to, typically because they require a certain level of access. For example, causing the server to access the user database directly for download.

We mitigate this by ensuring that there aren't any possible routes to sensitive files, and by validating any client inputs.
